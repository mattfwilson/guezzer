<objective>
UX-02: The screen wake lock leaks after End Show when `releaseWakeLock()` runs while an
`await navigator.wakeLock.request("screen")` is still in flight. The release no-ops on the
`null` sentinel, then the late-resolving request stores a live lock as `sentinel` that
nothing ever releases (`showActive` is already `false`) ã the screen stays awake until the
app is backgrounded.

Fix (D-02, locked): after the `await` resolves in `acquireWakeLock`, re-check `showActive`;
if it is already `false` (End Show fired during the in-flight request), release `next`
immediately and return ã do NOT store it as `sentinel`, do NOT call `onUnsupported`.

Purpose: Reliably release the wake lock after End Show even when release races an in-flight
acquire, without breaking the never-throw / silent-fallback browser-API contract.
Output: A re-check in `acquireWakeLock` + a component regression test for the race.

**Resolved decision (phase_shape open item):**
- **UX-02 rapid-restart residual (Open Question 2): ACCEPT & DOCUMENT ã do NOT add the epoch
  token.** The boolean `showActive` re-check (D-02) closes the reported End-Show leak. It does
  NOT close the rarer rapid End-ShowãStart-Show-while-old-acquire-in-flight orphan (A5), because
  a boolean cannot distinguish "show 1" from "show 2". That residual is LOW severity, has a tiny
  practical window, and self-clears on next background. Per the smallest-possible-hardening
  posture (carried from Phases 9 & 12) we accept it rather than expand scope beyond D-02 with the
  optional monotonic-`epoch` token. Document the residual with a code comment at the re-check so
  a future reader knows it is a conscious accept, not an oversight.
</objective>

# Phase 13 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase posture (carried from Phases 9 & 12): smallest-possible hardening, robustness over strictness, don't touch working UX beyond the four fixes.** This phase introduces **no new UI, no new components, no new tokens, and no new user-facing copy.** All four fixes (UX-01/02/03/04) operate *within* the app's already-shipped design system. This document therefore (1) records the existing contract these fixes must preserve, and (2) adds the four phase-specific *interaction* contracts. The token tables below are the established v1.0/v1.1 system quoted verbatim from `packages/app/src/styles.css` `@theme` and `packages/app/src/config.ts` — reproduced so the checker/auditor can confirm the fixes stay inside it, not because any new value is being declared.

## Spacing Scale

The app is built on a 4px base (Tailwind default `rem` scale, 1 unit = 4px). Multiples-of-4 hold across the codebase. Values relevant to this phase's surfaces:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline padding |
| sm | 8px | Compact element spacing |
| md | 12px | Header vertical padding — the `env(safe-area-inset-top) + 12px` header idiom (UX-01) |
| lg | 16px | Default element spacing |
| xl | 24px | Section padding; RecapView header inset `env + 24px` (UX-01 consistency decision) |
| 2xl | 32px | Layout gaps |
| 3xl | 48px | Major section breaks |

Exceptions (this phase):
- **Safe-area insets are additive, not fixed tokens.** Top-anchored surface headers use `padding-top: calc(env(safe-area-inset-top) + 12px)`. UX-01's contract is that `env(safe-area-inset-top)` is applied **once per surface** (per-surface `env()` idiom) — never inherited from `body` AND re-added. See Interaction Contract UX-01.
- **RecapView header** currently uses `env + 24px` (12px more than the other six top surfaces). This is a pre-existing, likely-intentional value; if planning normalizes it to `+12px` for header-height parity, that is a spacing decision, not a bug fix — document it either way (RESEARCH Open Question 1).

## Color

Verbatim from `styles.css` `@theme` (dark-only palette, no light theme, no toggle). 60/30/10 split is already enforced app-wide.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0C0C10` (`--color-surface`) | App background, all surfaces |
| Secondary (30%) | `#17171F` (`--color-elevated`) | Tab bar, sheets, banners, toasts, cards |
| Accent (10%) | `#F2C14E` (`--color-accent`) | See reserved list below |
| Destructive | `#EF4444` (`--color-destructive`) | Destructive confirmations only (e.g. unmark / delete-entry confirm) |

Supporting tokens (unchanged, not part of the 60/30/10 accounting): text-primary `#F5F5F7`, text-muted `#A1A1AA`, hairline `#2A2A34`.

Accent reserved for: **Install / Refresh (SW update) prompts, the primary phase CTA (Share), the pre-opener "Search for the opener" prompt orb, and focus rings — ONLY.** Never applied to all interactive elements. Rarity semantics use their own separate hue set (`config.dex.tierColors`: debut `#A1A1AA`, common `#E4E4E7`, uncommon `#34D399`, rare `#60A5FA`, epic `#A855F7`, legendary `#FB923C`) and the GizzVerse constellation uses tuning-family colors — data semantics are never expressed in accent chrome.

**Phase-13 color contract: no new color is introduced, and none of the four fixes may change any rendered color.** UX-04 must preserve the constellation's existing tuning-family + depth-shaded palette untouched (camera-only change). UX-03 changes hint *presence*, not color. The checker should confirm zero new hex values entered the codebase.

## Copywriting Contract

**This phase adds no new copy.** UX-01/02/04 are invisible layout/lifecycle/camera fixes with no text. UX-03 *reuses* existing live-sync copy and only suppresses it when ambiguous. The table records the existing strings these fixes operate against (all live in `config.copy.*`, read by components, never hardcoded — the established convention).

| Element | Copy |
|---------|------|
| Fill-hint eyebrow (UX-03, existing) | `Fill in ???` (`config.copy.live.fillPlaceholderEyebrow`) |
| Fill-hint body (UX-03, existing) | `Editor logged {songName} here — tap to fill it in.` (`config.copy.live.fillPlaceholderBody`) — shown ONLY when the placeholder's neighborhood unambiguously brackets the editor row; suppressed entirely on ambiguity (no wrong-song name ever surfaced) |
| Wake-lock fallback (UX-02, existing — must NOT fire on End-Show release) | `Keep your screen on manually — auto screen-wake isn't supported on this device.` (`config.copy.show.wakeLockFallback`) |
| Primary CTA (unchanged) | Share (accent, `Share2` icon) — GizzDex header + RecapView |
| Empty / error / destructive states | none introduced or altered this phase |

Copywriting rule for this phase: **UX-03 must never render `fillPlaceholderBody` with an incorrectly-matched `{songName}`.** A missing hint is the worst acceptable outcome; a confident wrong song name is a contract violation (D-03).

## Interaction Contracts (phase-specific)

These are the load-bearing contracts the executor must satisfy — the "interaction" half of the design contract, since there is no new visual surface.

### UX-01 — Safe-area inset: one idiom, per-surface `env()`
- **Contract:** `env(safe-area-inset-top)` is applied **exactly once** on the path from viewport to any top-anchored header. Remove the `body { padding-top: env(safe-area-inset-top) }` at `styles.css:186`; every top surface owns its own `calc(env(safe-area-inset-top) + Npx)`.
- **Visual acceptance:** on a notched iPhone installed PWA, a single `env` dead band above the header (no doubled ~50px band), and header content vertically aligned across all seven top surfaces: AppShell header, SearchSheet, ArchiveBrowser, RecapView, AlbumDetail, CompareView, SetlistView.
- **Constraint:** the six `fixed inset-0` overlays are already single-inset and correct — do NOT edit their `env()` calcs (RESEARCH Pitfall 1). The only per-surface *decision* is RecapView's `+24px` (keep as documented exception, or normalize to `+12px`).

### UX-02 — Wake-lock lifecycle: released after End Show
- **Contract:** after End Show, the screen wake lock is reliably released even when release races an in-flight acquire. The never-throw / silent-fallback browser-API contract is preserved.
- **Acceptance:** the End-Show release path releases any late-resolving lock and **must not** surface `wakeLockFallback` (End Show is normal teardown, not an unsupported device — RESEARCH Pitfall 2). Screen sleeps after End Show on-device.

### UX-03 — Fill-hint: never name the wrong song
- **Contract:** a fill-hint is offered only when the placeholder's surrounding logged songs unambiguously bracket exactly the right editor row(s); otherwise no hint. Pure-core suppression, before the UI.
- **Acceptance:** a position-gapped trail (skipped or mid-trail-deleted entries) yields the *correct bracketed* hint or *no* hint — never an off-by-N song name. All four existing `resolvePlaceholders` tests stay green.

### UX-04 — Constellation camera belongs to the user
- **Contract:** the constellation preserves the user's exact pan/zoom across container resizes (address-bar collapse, orientation, keyboard). `zoomToFit` fires only on the first settle per `graphData` change (genuine view switch), never on pure size changes or filter toggles. Settle-and-freeze (EXPL-06: frozen `fx/fy`) is preserved.
- **Exception:** if a *focused* node would fall off-screen after a resize, smoothly **pan** (at current zoom) to re-center it — never fit-all, never re-zoom (RESEARCH recommendation; Open Question 3). Any new constant (`FOCUS_OFFSCREEN_MARGIN_PX`) lives in `config.explore` — no scattered magic numbers.
- **Acceptance:** panned/zoomed into the graph, collapsing the address bar or rotating leaves the camera put; driving a focused node off-screen via resize re-centers it without fitting-all.

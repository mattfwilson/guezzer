## Shared Patterns (greenfield conventions — every file follows these)

No shared code exists yet; these come from CLAUDE.md + RESEARCH.md and apply project-wide from the first commit:

### Module system & imports
**Apply to:** every `.ts` file in `packages/core`
- ESM only (`"type": "module"`); **all relative imports carry `.ts` extensions** (Node native type stripping does no resolution magic — Pitfall 4)
- No `enum`, no `namespace`, no parameter properties (`erasableSyntaxOnly`)
- Core imports nothing from React/DOM/browser — enforced by tsconfig `lib` and absent deps, not by review

### Error message convention
**Source:** RESEARCH.md Pattern 2 (the `assertFilterApplied` example) + CONTEXT.md Specifics ("failure UX matters")
**Apply to:** `validate.ts`, `api-types.ts` locked schemas, `normalize.ts` contiguity assertion, all CLI failures
Every hard failure names: **endpoint/field, expected vs. actual value, and an example show** (id + date, or first 200 chars of the row). This convention is set in Phase 1 and copied by all later phases.

### Constants
**Apply to:** all core modules
Zero magic numbers/strings outside `packages/core/src/config.ts` — includes `apiBase`, `fetchDelayMs` (~2000), year range, User-Agent (`"Guezzer setlist tool (matt.f.wilson@gmail.com)"`), settype allowlist (locked post-census), sentinel `song_id` list (`[1]`).

### CLI structure
**Apply to:** all `cli/*.ts`
Thin orchestrators: parse/validate args (year as bounded integer 2010–2100) → call pure `ingest/*` functions → write files. All pacing/orchestration in Node, never shell (Windows dev machine). Runnable as `node packages/core/src/cli/refresh.ts` with no build step.

### Test structure
**Apply to:** all `test/*.test.ts`
Vitest `node` environment; fixtures are small real-show extracts imported as plain JSON; assertions against known expected outputs (CLAUDE.md testing constraint).

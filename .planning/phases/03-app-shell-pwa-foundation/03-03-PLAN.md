<objective>
Deliver the update-safety vertical slice of PWA-02: the concrete mechanism behind "the app never swaps versions mid-show." A waiting service-worker update surfaces as a non-blocking, dismissible toast that applies the new version ONLY when the user taps Refresh ŌĆö and a visible version stamp in the menu gives the owner a debugging anchor when a friend reports a bug.

Purpose: `registerType: 'prompt'` (set in Plan 01) means a new SW installs but WAITS. This plan wires the `useRegisterSW` React hook so `needRefresh` drives a toast, and `updateServiceWorker(true)` (skipWaiting + reload) is the ONLY thing that swaps versions. The mental model: friends refresh between shows, never mid-show (D-06). The version stamp (`v<pkg> ┬Ę <sha> ┬Ę built <date>`) is injected at build time via Vite `define` ŌĆö no runtime fetch, works offline (D-07).

Output: `useRegisterSW.ts` wrapper (with the hourly online-guarded update check), the `UpdateToast` component, the `VersionStamp` component reading the build constants, the version stamp placed in `AppMenu`, the toast mounted in `App.tsx`, and a `version.test.tsx` component test.

Covers PWA-02 (prompt-based update flow + visible version stamp). Implements D-06 (user-tap-only update toast) and D-07 (build-time version stamp).
</objective>

<phase_goal>
**As a** friend attending a King Gizzard show, **I want to** install Guezzer to my home screen and have it load fully offline with no surprise version swaps or data loss, **so that** I can rely on it in a dark venue with no signal.

This plan delivers the never-swap-mid-show slice: after it completes, updates arrive only via a user-confirmed prompt and the running version is always identifiable.
</phase_goal>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| service-worker lifecycle ŌåÆ app | A newly deployed SW installs and waits; the app decides when to activate it. |
| build constants ŌåÆ rendered UI | `__APP_VERSION__`/`__GIT_SHA__`/`__BUILD_DATE__` are injected at build and displayed. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-08 | Tampering | Update swap (skipWaiting) | mitigate | The new SW activates ONLY via `updateServiceWorker(true)` behind an explicit user Refresh tap (D-06). No `autoUpdate`, no auto-activation ŌĆö this is the exact "never swaps mid-show" control. workbox-window guards the controllerchange double-reload edge case. |
| T-03-09 | Tampering | XSS via version constants | mitigate | `__GIT_SHA__`/`__BUILD_DATE__`/`__APP_VERSION__` are build-time literals rendered as React text, never assigned to `innerHTML` or `eval`. No runtime version fetch to poison. |
| T-03-10 | Info Disclosure | SW registration error logging | mitigate | `onRegisterError` logs only the error object to console; no secrets/PII exist in this app to leak. |
| T-03-11 | DoS | Periodic update check | accept | Hourly `r.update()` is guarded by `navigator.onLine`; low frequency, same-origin, no third-party endpoint. Negligible. |

Inherits T-03-SC from Plan 01 (no new installs). No `security_block_on: high` items. Surface remains LOW (RESEARCH ┬¦Security Domain).
</threat_model>

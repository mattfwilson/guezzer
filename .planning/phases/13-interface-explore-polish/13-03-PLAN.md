<objective>
UX-03: `resolvePlaceholders` (suggest.ts:144-161) matches a trail `???` placeholder to a
`latest` editor row by raw `row.position === entry.position` (:151). But trail
`TrackedEntry.position` is monotonic max+1 (it GAPS on skipped/deleted entries — db.ts:325-334)
while editor `LatestSetlistRow.position` is contiguous 1,2,3. After the first divergence a
placeholder confidently names an off-by-N song, one tap from applying it via `renameEntry`.

Fix (D-03, locked posture): conservative-suppress — no hint beats a wrong hint. Only offer a
fill-hint when the placeholder's neighborhood unambiguously brackets exactly the right editor
row(s); otherwise emit nothing. This is a pure-core change in `packages/core` (zero DOM, zero
db import — CLAUDE.md strict separation).

Purpose: Fill-hints name the correct song even after skipped/deleted trail entries — never
an off-by-N — while keeping all four existing tests green.
Output: Rewritten `resolvePlaceholders` (unchanged `FillHint` interface) + new regression fixtures.

**Resolved decision (UX-03 discretion): interval-count-match subsequence anchoring** (RESEARCH
recommendation A2, hand-verified against all four existing tests):
1. Sort trail by `position`; sort editor rows by `position`.
2. Anchors = logged trail entries (`!isPlaceholder && songId !== null`), matched to editor rows
   **by `songId`** (the same by-song_id keying `diffLatestAgainstTrail` uses). Require anchors to
   form a strictly-increasing in-order subsequence of editor indices.
3. Partition both sequences into intervals bounded by consecutive anchors — INCLUDING the head
   interval (before the first anchor) and the tail interval (after the last).
4. Per interval: if `#placeholder-slots === #editor-rows-in-interval`, map 1:1 in order → emit
   `FillHint`s from the BRACKETED editor rows. Otherwise SUPPRESS that interval. Any
   out-of-order / absent-from-editor anchor → suppress the affected interval(s) (safe fallback:
   suppress everything). This keeps `FillHint.entryPosition = placeholder.position` (the UI's
   `renameEntry` handle) and takes `position/songId/songName` from the bracketed row, not the
   raw same-position row.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Realtime broker → engine callbacks | presence:sync state and broadcast:wave payloads enter through the engine's `.on(...)` callbacks — untrusted; validated/reduced before any store write or toast. |
| session lifecycle → channel | Sign-out / offline transitions must fully tear the channel down or a departed user keeps broadcasting as online. |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-01 | Tampering / Input Validation (V5) | engine broadcast:wave callback | mitigate | Every inbound wave passes `validateWave(raw, userId)` before `showWaveToast`; malformed/unknown-emoji/self/other-targeted → dropped silently, host never crashes (V7). |
| T-20-10 | Repudiation / stale presence (Pitfall 6) | engine teardown | mitigate | On sign-out/offline the `[userId, online]` effect cleanup calls `removeChannel` + `setWaveSender(null)` + `resetPresenceState()` so a departed/borrowed identity stops broadcasting and its dots clear (D-16/17/20). |
| T-20-03 | Information Disclosure | engine `.track(activity)` | mitigate | Tracks only `deriveActivity(...)` = `{tab, atShow?:boolean}`; the active-show liveQuery contributes a boolean, never a song (D-03 scope line). |

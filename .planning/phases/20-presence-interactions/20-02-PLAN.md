## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| peer wave payload → toast host | Inbound `WaveToastPayload` (already validated upstream by validateWave) still renders untrusted strings; the sender name is re-resolved from the trusted store, not the payload. |
| local user → Realtime broadcast | The palette emits a wave the user composed; the emoji is constrained to the fixed 4-set. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-20-06 | Tampering (XSS) | WaveToast sender line / ReactionPalette target rows | mitigate | All sender/emoji/displayName strings render as escaped React text; never `dangerouslySetInnerHTML`; names `truncate min-w-0` clamped. Unit test asserts a markup-bearing name is not injected. |
| T-20-02 | Spoofing | WaveToast sender-name resolution | mitigate | Sender name resolved from the trusted `getSyncState().friends` by `from` userId, NOT from the payload; unknown `from` renders a neutral escaped fallback (spoofed-from residual is LOW for a private ~5-friend tool). |
| T-20-04 | Denial of Service (wave-flood) | WaveToast FIFO queue | mitigate | Bounded FIFO with `config.presence.QUEUE_CAP` and over-cap DROP bounds on-screen flooding regardless of send rate (D-08 accepts no send limit). Unit-tested. |
| T-20-09 | Denial of Service (tap-blocking) | WaveToast host layer | mitigate | Host is `pointer-events-none` at `config.ui.z.toast` and registers its height so it never covers/intercepts the live-log loop (the sacred D-17 Bingo rule). |
| T-20-SC | Tampering | npm/pip/cargo installs | accept | No new dependencies (RESEARCH audit = N/A). No install task. |
</threat_model>

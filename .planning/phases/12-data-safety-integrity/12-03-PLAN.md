<objective>
SAFE-02: backup and share-card downloads must complete reliably on iOS Safari — no same-tick
`revokeObjectURL` aborting the download. Root cause: the anchor-download idiom was copied, so
the bug exists twice — `exportDownload.ts:60` and `shareCard.ts:286` both revoke the object
URL in a `finally` on the same tick as `anchor.click()`.

Fix (D-06/D-07): centralize the idiom into ONE `triggerDownload(data, filename)` helper in
`packages/app` (browser-only → NOT core) that defers the revoke via
`setTimeout(() => URL.revokeObjectURL(url), config.dataSafety.OBJECT_URL_REVOKE_DELAY_MS)`.
Both call sites adopt it, so the fix lives in one place and can't drift. The delay constant
lives in the single app config file (no scattered magic numbers). NOTE: `shareCard.ts`'s
`previewUrl` (~L234) is NOT a leak — it is already released by `ShareCardSheet`'s effect
cleanup — DO NOT "fix" it (12-RESEARCH.md §SAFE-02).

Purpose: the iOS-eviction backstop backup (and the share card) must actually reach the user's
downloads on their phone.
Output: the config constant, the new helper + its fake-timer test, and both consumers rewired.
</objective>

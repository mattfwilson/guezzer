<objective>
UX-04: The constellation camera snaps to fit-all on any container resize. The spacing effect
(ConstellationCanvas.tsx:221-233) is keyed on `[graphData, size.width, size.height]` and calls
`d3ReheatSimulation()`; with frozen `fx/fy` the reheat is visually inert but still runs
`cooldownTicks` and re-fires `onEngineStop` (:695-714), which UNCONDITIONALLY calls
`zoomToFit` (:707). So every iOS address-bar collapse / orientation / keyboard resize yanks the
camera back to fit-all mid-exploration.

Fix (D-05, locked): gate `zoomToFit` behind a first-settle flag so it fires ONLY on the initial
engine stop per `graphData` change — never on pure size changes. Preserve the user's exact
pan/zoom on resize. Exception: if a FOCUSED node would fall off-screen after the resize, smoothly
re-center on it (do NOT fit-all).

Purpose: The camera belongs to the user — a resize is not a request to reframe.
Output: A `firstSettleRef` gate + an off-screen focus re-center + one new config constant.

**Resolved decisions (phase_shape open items):**
- **UX-04 re-center: pan-only, NOT re-zoom (Open Question 3).** On the off-screen exception,
  keep the current zoom `k` and PAN (`centerAt`) to bring the focused node back — the smallest
  intervention that respects the user's chosen zoom (A7). Do NOT snap to `FOCUS_ZOOM_K`. Reuse
  the existing `FOCUS_CAMERA_DURATION_MS` for the ease; add NO re-zoom constant.
- **First-settle key = `[graphData]` only (Pitfall 4).** Filters (`topK`/`overlay`/
  `visibleNodeIds`) provably never rebuild `graphData` (memo comments :253-259) — only a view
  change (Rotation ↔ Full catalog) does, which legitimately should re-fit. So key the reset
  strictly on `[graphData]`, never `[size]` or a filter dep.
</objective>

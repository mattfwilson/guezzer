# v2.1 — UX/UI Polish Backlog

Captured from the owner's Obsidian note during the Phase-20 close-out session. This is the
scope source for a future **v2.1 "UX/UI Polish"** milestone (decision: 2026-07-24 — do NOT
fold into the completed, verified Phase 20).

**Global constraint:** every change must function/render correctly on desktop AND mobile,
**mobile-first** (the August live shows are the end use case).

**None of these are show-blocking bugs** — all are enhancements. The one real functional

## Items (verbatim intent, tagged [F]=functional/UX, [UI]=visual)

### In-show / tracking experience  [F]
1. **Bingo deal stays in tracking.** When tracking a live show and prompted to deal a bingo
   card, keep the user *inside* the tracking experience instead of sending them to the
   GizzGames tab. Idea: overlay the bingo-dealing screen on top of the tracking UI.
2. **Bingo view from the FAB.** The expandable/collapsible in-show bingo view should be
   accessed from the FAB button → opens a **full-screen overlay** on top of tracking so the
   user can view progress, then dismiss to return to tracking.
3. **Bingo toast deep-link.** Bingo toast notifications get a button to jump straight to the
   bingo-card overlay in tracking. *(Sequence: do after #2.)*
4. **Hide tabs while in-show.** When a show is being tracked ("in-show"), the bottom tabs
   should not be visible/available — maximize immersion.

### Navigation / chrome
5. **Rename bottom tabs** (less redundant): LiveGizz→**Live**, GizzVerse **stays**,
   GizzMap→**Map**, GizzDex→**Dex**, GizzGames→**Games**.  [UI]
8. **Relocate install affordance.** Remove the "Install Gizz With Friends" button from the
   top-right menu; move the 3 add-to-home-screen steps to the **bottom of Settings**.  [UI]
9. **Modular fullscreen toggle.** In GizzVerse, a fullscreen button that animates away the
   top menu bar + bottom tabs for more constellation viewing area, easily toggled back.
   **Build it modular** — the same capability will be reused in other sections.  [F]

### Reactions
6. **Reactions fly up from the sender's section.** Instead of a toast, when a user sends a
   reaction, the emoji + their name should randomly **fly/animate upward from whatever
   section the sender is currently on** — more alive, and it conveys which part of the app
   they're reacting from.  [F]

### Small visual
7. **Bingo deal-type icons.** On the deal-a-card screen, add fun icons to the left of each
   difficulty: *Deal Chill* (easy), *Deal Balanced* (medium), *Deal Glory-hunter* (hard).  [UI]

## Open decisions to resolve at milestone kickoff

- **#6 reworks shipped Phase-20 code.** The current reaction receive-display is the
  `WaveToast` FIFO toast (`packages/app/src/components/WaveToast.tsx`). #6 replaces that
  *presentation* with a section-anchored fly-up. The send surface (`ReactionPalette`),
  `validateWave`, and the `gizz-room` transport all stay; only the display changes. It
  composes with presence (we already track the sender's tab). **Tension:** it's
  motion-forward vs. the app's reduced-motion / settle-and-freeze discipline (same reason
  "GizzVerse directional-flow particles" was parked) — needs a reduced-motion fallback and
  an explicit owner OK to lean into the animation.
- **#5 partially reverses rebrand `260716-wwj`** (which moved *toward* GizzDex/GizzMap/
  GizzGames). #5 walks Map/Dex/Games back to bare names but keeps GizzVerse. Confirm the
  asymmetry is intended, and check for any user-facing copy that references the long names.
- **#9 modularity** — implement as a shared hook/context (e.g. an "immersive/chrome-hidden"
  mode) so GizzVerse and future sections reuse it; note it overlaps conceptually with #4
  (hide tabs in-show) — both are "hide app chrome," so a single shared mechanism may serve
  both.

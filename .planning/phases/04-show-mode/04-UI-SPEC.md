# Phase 4 — UI Design Contract

> Visual and interaction contract for Show Mode: the one-thumb live tracking loop. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This phase INHERITS the Phase 3 foundation** (`.planning/phases/03-app-shell-pwa-foundation/03-UI-SPEC.md`): spacing scale, typography, dark-only chrome palette, 44px tap floor, `lucide-react` icons, system font. Tokens are **not re-derived here** — they are extended. New for Phase 4: the orbit stage, tuning-family orb palette (ratified from the Phase 3 forward declaration), the hit/miss semantic status pair, the comet trail, the bottom action bar, and Show-Mode copy/state contracts.

## Spacing Scale

Inherited verbatim from Phase 3 (multiples of 4, Tailwind v4 4px base). **Do not re-declare in code — reuse the existing `@theme` scale.**

| Token | Value | Usage (Phase 4 additions) |
|-------|-------|---------------------------|
| xs | 4px | Orb %-to-name gap, trail-ring stroke offset |
| sm | 8px | Action-bar inter-button gap, trail-node gap |
| md | 16px | Action-bar padding, search-sheet gutters, orbit stage inset |
| lg | 24px | Bottom action-bar vertical block, dialog padding |
| xl | 32px | Pre-show launcher vertical rhythm |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Pre-show empty-state top spacing |

**Exceptions (functional, inherited + extended):**
- **44px minimum interactive target** — hard floor for every tappable element. In Show Mode this is load-bearing: **every orb is ≥ 44px regardless of its probability** (SHOW-02), every action-bar button, every trail node (including the "+N" chip), and the info/"why" affordance. Apply via `min-h-11 min-w-11` + padding; never shrink below it even when a low-score orb would geometrically want to.
- **Orb minimum diameter = 56px** (recommended, config-tunable `ORB_MIN_DIAMETER`) so the ≥44px hit area comfortably contains an on-orb % + truncated name in the dark. The 44px floor is the absolute hit-area minimum; 56px is the visual minimum.
- **Safe-area insets** via `env(safe-area-inset-*)` — the bottom action bar MUST sit above the home-indicator inset; the orbit stage MUST NOT place any orb under a notch/inset.

## Typography

Inherited: **two weights only** (regular 400, semibold 600), **four sizes**. No new sizes introduced. Phase 4 usage mapping:

| Role | Size | Weight | Line Height | Phase 4 usage |
|------|------|--------|-------------|---------------|
| Body | 16px | 400 | 1.5 | Search result rows, dialog body copy, weak-fan hint |
| Label | 14px | 600 | 1.4 | Action-bar button labels, orb song name (truncated), orb % (with `tabular-nums`), tally readout, trail-node caption |
| Heading | 20px | 600 | 1.2 | Center current-song name, dialog headings, pre-show launcher heading |
| Display | 28px | 600 | 1.2 | Reserved — not required in Show Mode; center song stays at Heading to leave room for long KGLW titles |

Notes:
- **Orb %** renders with `tabular-nums` so digits don't reflow between repredicts (`0%`–`97%`). Format: `round(score*100)%`; if it rounds to `0`, show `<1%` (never a bare `0%` on a shown orb).
- **Tally** renders with `tabular-nums`: `{hits}/{total} · {pct}%`.
- Orb song name truncates with ellipsis to a single line; the full name is available via the orb's info affordance and the center node when tapped-to-center.
- Body never below 16px (dark-venue readability; iOS form-zoom avoidance in the search field).

## Color

Dark theme is **mandatory** (inherited — no light theme, no toggle). Two distinct color axes below: (A) the 60/30/10 **chrome** palette (inherited), and (B) two Show-Mode **semantic/data** axes that are NOT part of the 60/30/10 split.

### A. Chrome palette (60/30/10 — inherited, unchanged)

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0C0C10` | Orbit-stage background, app background |
| Secondary (30%) | `#17171F` | Elevated surfaces: bottom action bar, comet-trail strip, tally chip, search sheet, dialogs, center-node backing |
| Accent (10%) | `#F2C14E` (warm gold) | See reserved list |
| Text primary | `#F5F5F7` | Center song name, orb name/%, tally, labels |
| Text muted | `#A1A1AA` | Secondary/disabled copy, weak-fan hint, trail captions |
| Border / hairline | `#2A2A34` | Action-bar top edge, dialog separators, focus-ring base |

**Accent gold reserved for (explicit — never "all interactive elements"):**
1. The **Start Show** primary CTA fill (pre-show launcher) — the single most important pre-show tap.
2. The **focus-visible ring** on keyboard/AT focus (inherited).
3. The active/pressed state of the **info "why"** affordance while its detail is open.

Everything else — Search, ???, Set break, Encore, Undo, trail nodes — uses text-primary/text-muted on the secondary surface. Orbs are colored by the **data palette (B)**, never accent. Accent scarcity keeps "Start Show" unmistakable, and prevents the venue's most-tapped controls (Search/???) from competing with orbit color coding. **Accent gold is deliberately NOT used for orbs**, because the microtonal family color derives from gold-shifted coral and must stay visually separate from chrome accent.

### B1. Tuning-family DATA palette (ratified from Phase 3 forward declaration)

Colors the orbs (SHOW-01) and the center current-song node, keyed to the three families in `data/tuning-tags.json` via `MatrixNode.tuningFamily`. **Ratified here as the Phase-4 contract** (Phase 3 declared these provisional pending this spec). Not chrome — never used for buttons/surfaces.

| `tuningFamily` value | Value | Notes |
|----------------------|-------|-------|
| `standard` | `#5EC8E5` (cyan) | Most of the catalog |
| `C# standard` | `#B98CF2` (violet) | Third family per CLAUDE.md |
| `microtonal` | `#FF8A5B` (warm coral) | Gold-shifted; kept distinct from chrome accent gold |

Requirements this palette MUST satisfy (checker-verifiable): color-blind-safe separation across the three, and **≥ 3:1 contrast against `#0C0C10`** for the orb fill/ring (non-text graphical contrast, WCAG 1.4.11). On-orb text (name/%) is `#0C0C10` dark-on-orb OR `#F5F5F7` per-family whichever clears **4.5:1** on that family's fill — executor picks per family and records it. A song with an unmapped/missing tuning family falls back to text-muted `#A1A1AA` fill (neutral, never invented color).

### B2. Hit / miss semantic status pair (NEW this phase)

Distinct from both chrome and the tuning palette. Drives the comet-trail rings (SHOW-08), the tally, and confirm-logging feedback. This is the "second semantic color" the contract allows, extended to a hit/miss pair because scoring is core to Show Mode (D-06/D-08).

| Role | Value | Usage |
|------|-------|-------|
| Hit | `#22C55E` (green) | Trail-node ring when the confirmed song WAS in the shown fan (D-06); positive tally tint (optional) |
| Miss | `#EF4444` (red — reuses inherited Destructive) | Trail-node ring when logged via Search/??? / off-catalog (D-08); also the Destructive action color below |

**Destructive** (SHOW-07, End Show) reuses `#EF4444`. Miss-ring red and destructive-action red are the same hue by design — both mean "the model didn't get this / removal." Both MUST clear ≥ 3:1 against `#0C0C10` (ring) and destructive-button text ≥ 4.5:1. Do not add a third red.

Weak-fan softening (D-10): when the top orb's `score < WEAK_FAN_THRESHOLD` (config, default `0.15`), reduce orb fill opacity to ~55% and desaturate, and show the muted "Low confidence" hint (copy below). No number is faked; the honest small % still renders.

### Destructive actions (SHOW-07, D-04, D-15)

| Action | Confirmation approach |
|--------|----------------------|
| **Undo last song** (D-15) | **No dialog** — one tap removes the most recent entry immediately (the common "oops"); a brief inline "Removed {song} · Undo" affordance MAY allow re-adding, but is optional. Fastest-path-first per D-14/D-15. |

`???` logging (D-14) and orb-tap logging (SHOW-03) are the **fastest-must-be** paths and carry **no confirmation** — a plain tap on an orb LOGS it (never opens the "why"; that's the separate `Info` affordance, D-11).

## Layout & Interaction Contract

Region stack, top → bottom (all within the existing `#/show` hash route, inside the Phase 3 `AppShell` chrome):

1. **Header row (extends AppShell):** app wordmark + menu (inherited) on the left; **date stamp** (auto-stamped `today`, D-01/D-05) and the persistent **tally** (`12/18 · 67%`, SHOW-09) top-right; **End Show** control accessible here (icon + label). The tally is text-primary, never accent.
2. **Comet-trail strip (SHOW-08):** a horizontal band below the header on the secondary surface. Shows the **last ~4 songs** as diminishing nodes (nearest the orbit = most recent), each wearing a **hit ring (green) / miss ring (red)** (D-06/D-08). Older history compresses into a tappable **"+N"** chip once the set reaches **30+ songs** (`TRAIL_COMPRESS_AT`, config, default 30) — tapping "+N" expands the full setlist in a scrollable sheet. Tapping any node → edit/delete (D-15) or rename-??? (D-14). Must not crowd the prediction fan; the strip is fixed-height and horizontally scrollable, never wrapping into the orbit.
3. **Orbit stage (SHOW-01/02/03/10 — the focal region):** fills the remaining vertical space. **Center** = current song node (tuning-family colored, Heading-size name, no %). **Fan** = adaptive **5–8 orbs** (D-12) placed by a **deterministic radial layout** (never a force simulation — hard constraint, CLAUDE.md): higher score → nearer center + larger diameter; angle assigned deterministically by rank so tap targets never move on their own (SHOW-02). Each orb shows **truncated song name + %** and a small `Info` "why" dot (D-11). Orbs never below 44px hit-area / 56px visual (see Spacing). Weak-fan softening applies per D-10.
4. **Bottom action bar (D-13 — persistent, thumb-reachable):** fixed above the safe-area inset, on the secondary surface, two rows per the user-endorsed mockup:
   ```
   ── orbit fan ──
   [🔍 Search] [??? Unknown]
   [Set break] [Encore] [↶ Undo]
   ```
   Primary row: **Search**, **???**. Secondary row: **Set break**, **Encore**, **Undo**. Nothing overlaps the fan; nothing floats around the orbit (mis-tap avoidance). Every button ≥ 44px.

**Interaction rules:**
- **Tap orb** → log as hit + append old current to trail + recenter on the tapped orb + recompute predictions (SHOW-03). No confirm. Radial layout re-lays-out deterministically for the new center.
- **Tap `Info` dot on orb** → open "why" detail (SHOW-10) using `PredictionCandidate.reason`; does NOT log (D-11).
- **Tap Search** → full-screen/large search sheet, fuzzy over the full catalog via core `searchCatalog(query)` (fuse.js, wrapped); selecting a result logs a **miss** + recenters (D-06). Search must feel as fast as tapping a hit (the miss path is the fastest-must-be path).
- **Tap ???** → immediately append a `???` placeholder node + recenter; **miss** ring; no confirm (D-14). Renameable later from the trail (D-15).
- **Set break / Encore** → mark set structure, serialized to round-trip with kglw.net encoding (`setnumber`, `"e"` = encore; `docs/SCHEMA.md`, SHOW-06). Does not end the show (D-04).
- **Undo** → remove most recent entry, one tap, no dialog (D-15).
- **Force-quit → relaunch** → auto-resume straight into the single active show (D-03/SHOW-11); every confirmed song is write-through to IndexedDB immediately (Dexie `version(2)`, additive).
- **Gesture suppression (SHOW-13):** disable pull-to-refresh / overscroll bounce, text selection on tap targets, and double-tap zoom over the orbit stage and action bar (`touch-action: manipulation`, `overscroll-behavior: none`, `user-select: none` on controls). The orbit stage must not scroll or rubber-band under a fat thumb.
- **Motion:** CSS transitions only for recenter/orb re-layout (respect `prefers-reduced-motion`); `motion` lib only if the recenter choreography demands it (CLAUDE.md — start with CSS). Orbs animate to their new deterministic positions; positions themselves never drift (no physics).
- **Contrast/a11y floor (inherited):** all text ≥ 4.5:1 on its surface; orb fills ≥ 3:1 (graphical); focus-visible ring on all interactive elements.

## Config surface (single-config-file ethos — `packages/app/src/config.ts`)

Show-Mode tunables the executor MUST add under a `show` key (no scattered magic numbers, CLAUDE.md). Defaults are Claude's-discretion starting points (D-10/D-12 explicitly config-tunable):

| Key | Default | Meaning |
|-----|---------|---------|
| `ORB_COUNT_MIN` / `ORB_COUNT_MAX` | 5 / 8 | Adaptive fan bounds (D-12) |
| `ORB_DROP_SCORE` | 0.02 | Drop orbs below this score before hitting MIN (D-12) |
| `WEAK_FAN_THRESHOLD` | 0.15 | Top-orb score below which the fan softens (D-10) |
| `ORB_MIN_DIAMETER` | 56 (px) | Visual min; hit area stays ≥44px |
| `TRAIL_VISIBLE_RECENT` | 4 | Diminishing nodes before compression (SHOW-08) |
| `TRAIL_COMPRESS_AT` | 30 | Set length that triggers "+N" (SHOW-08) |

Copy strings above live under `config.copy.show.*`, mirroring the existing Phase 3 `config.copy` pattern.

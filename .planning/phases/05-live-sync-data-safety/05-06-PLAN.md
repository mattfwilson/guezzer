<objective>
Close the single FAILED must-have from Phase 5 verification (05-VERIFICATION.md): PWA-04 / SC4 / D-10 / T-05-07 — "never drop a local row on import". A wipe-then-restore round-trip works, but importing a backup into a NON-EMPTY database (the friend-as-second-set-of-eyes merge, and re-import onto a device that still holds data) silently DROPS rows.

Root cause (CR-01, Blocker): the core union-merge keys rows correctly by logical identity (sessionId, position) but each merged `trackedEntry` still carries its device-local Dexie `++id`. Two devices both number ids from 1, so the merged array contains COLLIDING ids across sources; `importSnapshot` commits via `bulkPut` (upsert by primary-key `id`), so colliding ids collapse last-write-wins — destroying a mix of local and incoming rows. `serialize.ts` also emits the volatile id into exported backups, so files carry per-device ids that collide on merge. The passing round-trip test `wipeAll()`s before every import, so ids never collide and the defect is invisible.

Also fix in the same pass (WR-01, Warning — a second data-loss path): the same-show dedupe tie keeps the INCOMING setlist, dropping a LOCAL-only setlist on a tie, contradicting D-10 "local survives".

Purpose: honour the D-10 / T-05-07 "never drop a local row" invariant that is the entire reason this phase exists — a friend's backup (or a re-import onto a populated device) must union, never clobber.
Output: id-less merge output, local-wins tie-break, logical-identity commit in `importSnapshot`, and a regression test that imports into a populated DB with overlapping ids and asserts every row survives.
</objective>

## Trust Boundaries

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05 | Tampering | `JSON.parse` in `parseAndMergeImport` | mitigate (already shipped) | Parse wrapped in try/catch; a syntax error rejects the whole file before any merge. Unchanged by this plan. |
| T-05-06 | Tampering | `exportEnvelope.safeParse` shape gate | mitigate (already shipped) | Strict `z.strictObject` rejects unexpected/`__proto__` keys before merge. Unchanged; `trackedEntryRow.id` stays optional so legacy files validate. |

Fix: commit trackedEntries by logical identity via `clear()` + `bulkAdd(id-less rows)` INSIDE the existing single rw transaction. The transaction is the key control — a mid-write throw rolls back the clear, so a failed import can NEVER leave the dex half-wiped (no partial state). Merge strips the volatile `++id` so no cross-device id collision can silently drop a row. |
| T-05-SC | Tampering | package installs | mitigate (n/a) | No new dependencies added; no npm/pip/cargo install tasks in this plan. |

**Key control documented:** the `clear()` + `bulkAdd()` pair for trackedEntries MUST remain inside the same `db.transaction("rw", ...)` block as the other three tables' `bulkPut`s. This preserves the D-12 no-partial-merge / atomic-rollback guarantee the phase already established — clearing trackedEntries outside the transaction would introduce a new half-wipe failure mode, so it is explicitly forbidden.

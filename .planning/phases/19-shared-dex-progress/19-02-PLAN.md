<objective>
Build the app-layer Supabase sync module under the `packages/app/src/sync/` fence and mount it app-wide. The genuinely-new mechanical work: identity-safe own-row upsert primitives (PROG-02), an app-wide `postgres_changes` subscription with a validated full re-pull (PROG-05), a reconnect flush + first-sync (D-17), and a Dexie offline last-known cache (D-18). Critically, the ENGINE that owns the subscription + the debounced upsert is a dedicated `useProgressSync()` hook **mounted once in `App.tsx`** (mirroring the shipped `useBingoCelebrations()` app-wide-mount precedent), gated internally on a signed-in identity ‚Äî NOT on the Friends toggle. `useFriendsProgress()` becomes a PURE read hook over a shared sync store, so the residency payoff (a friend's row moving live via PROG-05; the user's own dex upserting as they log a setlist via PROG-02) happens while the user is in Show Mode / LiveGizz, exactly per locked D-16.

Purpose: D-16 is explicit and locked ‚Äî the subscription is "active app-wide while signed in, not only while the Friends toggle is open." The prior wiring owned the engine inside `useFriendsProgress()`, whose only consumer (`FriendsList`) renders solely on the Friends sub-tab, so the engine died the moment the user left that tab. Splitting the engine from the read-hook and mounting it at the shell fixes this and makes it a singleton (no double subscription / double debounce).

Output: `progressSync.ts` (primitives + shared sync store), `useProgressSync.ts` (the app-wide engine), `useFriendsProgress.ts` (pure read hook + `buildFriendRows`), `friendCache.ts`, the `App.tsx` mount, the Dexie `version(8)` store, the config block, and mocked-supabase tests. Consumed by the Friends UI in 19-03.
</objective>

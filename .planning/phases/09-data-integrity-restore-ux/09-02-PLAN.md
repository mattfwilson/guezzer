## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user-selected backup file → import parser | Untrusted JSON (arbitrary file contents) crosses into `classifyImport`/`parseAndMergeImport` and, on the mine path, into Dexie writes |
| typed prompt input → merge routing | A free-text name decides whether an untrusted file is merged into local data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-09-04 | Spoofing | `isTypedNameMine` file-owner leg (anyone typing the file's owner name reaches merge) | accept | Personal offline tool for <10 friends, no accounts, no remote surface; the file is user-selected from local storage; matching the FILE's owner is the intentional WARNING-1 hardening — without it an evicted-DB owner cannot restore at all. Rationale doc-commented on the helper |
| T-09-05 | Tampering | Backup JSON parsing (`classifyImport` → zod envelope validation) | mitigate | Existing zod envelope schema validates before any write path is reachable (D-17 fork is structural); malformed JSON exercised by existing importFork tests; this plan adds no new parse surface |
| T-09-06 | Denial of Service (user-data loss) | `pickAndImport` merge on evicted DB | mitigate | Union merge committed in one Dexie transaction; Task 3's integration test proves seeded local rows survive and file rows are added — the specific data-loss regression PWA-05 guards against |
| T-09-07 | Information Disclosure | `ownerName` meta after import | mitigate | Owner stays a device-local fork key, never written to meta on import (06-07 decision); Task 3 asserts meta remains unset post-merge |
| T-09-SC | Tampering | npm/pip/cargo installs | accept | No new packages installed in this plan — supply-chain surface unchanged |
</threat_model>

<objective>
Deliver the data-durability vertical slice of PWA-03: stand up the real IndexedDB foundation (via Dexie) that every later feature-phase persists into, prove a genuine domain write survives relaunch, and request eviction-resistant storage — silently, without scary UI.

Purpose: This is the persistence proof for the phase goal's "never silently lose their data." A thin-but-real Dexie v1 schema (a `meta` settings table PLUS a stub `attendedShows` table keyed by the Phase-1-confirmed stable 10-digit `show_id`) proves PWA-03 with an actual domain write, not a generic key/value stub — and is shaped so Phase 4/5/6 grow it via additive versioned migrations without rework (D-08). `navigator.storage.persist()` is requested early and its result recorded in `meta` so a later phase can surface an export nudge; denial is silent because on iOS Safari `persist()` frequently returns false yet data still survives, and the real eviction backstop is Phase 5's JSON export (D-09, Pitfall 4).

Output: `db/db.ts` (Dexie subclass + typed tables + `setMeta`/`getMeta`), `pwa/persist.ts` (`requestPersistenceOnce`), the persistence request wired to fire early in `App.tsx`, and two unit tests (`db.test.ts` round-trip via fake-indexeddb, `persist.test.ts` status-recording + never-throws-on-denial).

Covers PWA-03 (IndexedDB persistence + `navigator.storage.persist()`). Implements D-08 (thin-but-real Dexie v1 schema) and D-09 (early, silent, status-recorded persistence request).
</objective>

<phase_goal>
**As a** friend attending a King Gizzard show, **I want to** install Guezzer to my home screen and have it load fully offline with no surprise version swaps or data loss, **so that** I can rely on it in a dark venue with no signal.

This plan delivers the never-lose-data slice: after it completes, personal data written to IndexedDB persists across relaunches and eviction-resistant storage has been requested.
</phase_goal>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app → IndexedDB | Personal data (attendedShows, meta) is written to client-side IndexedDB via Dexie; the only datastore (no backend). |
| storage API → app | `navigator.storage.persist()` governs eviction resistance; its result is recorded, not acted on with UI. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-12 | DoS (data loss) | IndexedDB eviction on iOS | mitigate/accept | `navigator.storage.persist()` requested early to reduce (not eliminate) iOS eviction risk; status recorded for a later export nudge. Residual risk is accepted here — the real backstop is Phase 5's prominent JSON export (PWA-04), out of Phase 3 scope. |
| T-03-13 | Tampering | Dexie schema / write path | mitigate | Access is via Dexie (typed tables, transactions) — no raw IndexedDB footguns; keys are `&`-unique. No external input is written in Phase 3 (attendedShows writes are local/test only; JSON import is Phase 5). |
| T-03-14 | Error handling | `persist()` denial | mitigate | Denial is caught and recorded silently; the function never throws (D-09) — no crash path, no error UI a user can't act on (Pitfall 4). |
| T-03-15 | Info Disclosure | IndexedDB data on a shared device | accept | Personal tool, no PII beyond show-attendance, no cross-user model; explicitly out of scope per project constraints (RESEARCH §Security Domain). |

<objective>
UX-01: On a notched iPhone installed PWA, the top safe-area inset must apply exactly
once. Today `body` (styles.css:186) applies `padding-top: env(safe-area-inset-top)` AND
the in-flow AppShell header re-adds `calc(env(safe-area-inset-top) + 12px)`, producing a
doubled ~50px dead band. The six `fixed inset-0` overlays escape body flow, so their
headers sit one inset HIGHER than the shell header today. Deleting the single body top-pad
line collapses everything to one per-surface `env()` idiom (D-01) and realigns all seven
headers automatically — no overlay edits.

Purpose: Remove the doubled inset and align overlay headers with the shell header without
touching working overlay code (smallest-possible hardening posture).
Output: One-line CSS deletion, a RecapView `+24px` decision documented in code, and the
phase's consolidated device-UAT checklist.

**Resolved decisions (phase_shape open items):**
- **UX-01 discretion (top-only vs full four-inset migration):** ship the **minimal top-only
  deletion** (delete only `styles.css:186`). Leave the L/R/bottom body insets on `body`.
  RESEARCH flags a full four-inset migration as an enlarged blast radius (must re-add
  L/R/bottom to `<main>` and re-verify every overlay) that risks the posture. Top-only fully
  fixes the reported doubled inset + overlay misalignment (A1).
- **RecapView `+24px` normalization (Open Question 1):** **LEAVE `env + 24px` as a deliberate
  exception**; do NOT normalize to `+12px` (normalizing changes recap layout — RESEARCH
  Pitfall 1 / A6). Mark it intentional with a one-line code comment so the 12px offset is
  self-documenting and never "corrected" later.
</objective>

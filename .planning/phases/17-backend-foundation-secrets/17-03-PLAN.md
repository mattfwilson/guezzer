<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| authenticated user → other users' rows | RLS must confine writes to `auth.uid() = user_id` |
| ops seed script (Node) → GoTrue admin API | service_role admin call must run server/ops-side only |
| realtime subscriber → progress rows | `postgres_changes` streams only rows the JWT's select policy allows |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-06 | Tampering / Elevation of Privilege | `public.progress` RLS | mitigate | `insert`/`update` gated `with check (auth.uid() = user_id)`; `select` read-all to `authenticated` only, NO `anon` policy (D-02); verified by cross-user write attempt in 17-04 |
| T-17-07 | Information Disclosure | `seed-users.ts` service_role handling | mitigate | Key read from `process.env` only (never `import.meta.env`/`VITE_`); roster carries no secrets; passwords env-keyed by slug (D-08) |
| T-17-08 | Information Disclosure | Realtime publication | mitigate | `select` policy `using (true)` to `authenticated` (intended friends-feed read-all); `anon` NOT granted, so unauthenticated `postgres_changes` cannot read (D-02/D-03) |
| T-17-SC | Tampering | npm installs | accept | Seed is dependency-free (global `fetch`, `node --env-file`); no installs in this plan |
</threat_model>

# Phase 15 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

**Phase posture: persistence/lock/replay plumbing + three small new surfaces inside the already-shipped design system.** Phase 15 delivers BINGO-06 (catch-up) and BINGO-07 (replay) plus the Dexie v5 / export-round-trip machinery. Most of the work is invisible (schema, lock state-machine, freeze, migration). The **only new visible UI** is: (1) a 4th bottom tab **"GizzGames"** with a replay list + "coming soon" teaser (D-01/D-02), (2) a **"Bingo" section inside the existing `RecapView`** — a read-only frozen board (D-05/D-06), and (3) a **catch-up surface** — a pre-checked confirm-list + manual search (D-03/D-04). The Deal/vibe/swap build UX, live marking, "one away" tension, and celebrations are **Phase 16** and must not be pulled forward.

All token tables below are the established v1.0/v1.1/v1.2 system quoted from `packages/app/src/styles.css` `@theme` and `packages/app/src/config.ts` — reproduced so the checker/auditor can confirm the new surfaces stay inside it. This phase introduces **no new color, no new font, no new component library, and no new spacing base**.

## Spacing Scale

The app is built on a 4px base (Tailwind default scale, 1 unit = 4px). Multiples-of-4 hold app-wide. Values relevant to this phase's three surfaces:

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap (tab), inline chip padding |
| sm | 8px | Bingo grid gap; square inner padding; confirm-list row gap |
| md | 12px | Top-header vertical inset idiom (`env(safe-area-inset-top) + 12px`) |
| lg | 16px | Section padding (GizzGames view, RecapView Bingo section, catch-up sheet) |
| xl | 24px | Section breaks between recap sub-sections; teaser card padding |
| 2xl | 32px | Layout gaps / empty-state vertical rhythm |
| 3xl | 48px | Major page-level spacing |

Exceptions (this phase):
- **Tap targets ≥ 44px** on every interactive element: the new **GizzGames tab** (reuses `min-h-11 min-w-11` from `BottomTabBar.tsx`), each **catch-up confirm-list row** (checkbox row), the **Add N** button, and any replay-list row. The 3→4 tab split is accepted (D-01); tabs stay `flex-1` so each remains ≥ 44px wide on the narrowest supported iPhone.
- **Bingo replay squares are read-only** (no direct-mark tap — D-04) so the 4×4 cells are sized by grid (`~80px` on a 360px viewport), not by the 44px tap-target floor. Only genuinely-interactive elements owe 44px.
- **Safe-area insets are additive, not fixed tokens.** The GizzGames view (a top-anchored tab surface) owns its own `calc(env(safe-area-inset-top) + 12px)` header inset, applied **once** per the UX-01 per-surface idiom — never inherited from `body`.

## Typography

Existing system type scale (system font stack). Weights: **regular 400 + semibold 600** (exactly 2). No new font, no new weight. Four roles are in play this phase:

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Square label (micro — bingo cell text, tab label) | 12–14px | 600 | 1.2 |
| Label / eyebrow (row copy, captions, "Lit by…") | 14px | 400 (muted `#A1A1AA`) | 1.4 |
| Body (teaser body, confirm-list song names, CTA) | 16px | 400 | 1.5 |
| Heading (section titles: "GizzGames", "Bingo", teaser title) | 20px | 600 | 1.2 |

Notes:
- The **bingo cell label** renders at 12px semibold on the densest cells (long square names like "Marathon jam" / song titles), wrapping to ≤2 lines with ellipsis; the tab label reuses the shipped `text-[14px] font-semibold` from `BottomTabBar.tsx`. Both sit in the 12–14px semibold micro role.
- The Display 56px role (GizzDex completion stat) is **not** used on any Phase-15 surface.
- Per-square "which song lit this" caption uses the 14px muted eyebrow role.

## Color

Verbatim from `styles.css` `@theme` (dark-only palette, no light theme, no toggle). 60/30/10 split is enforced app-wide.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0C0C10` (`--color-surface`) | App background, GizzGames view bg, unmarked-square/board bg base |
| Secondary (30%) | `#17171F` (`--color-elevated`) | Tab bar, teaser card, catch-up sheet, unmarked bingo squares, confirm-list rows |
| Accent (10%) | `#F2C14E` (`--color-accent`) | See reserved list below |
| Destructive | `#EF4444` (`--color-destructive`) | Not used this phase — no destructive action introduced |

Supporting tokens (unchanged, not part of the 60/30/10 accounting): text-primary `#F5F5F7`, text-muted `#A1A1AA`, hairline `#2A2A34`, **caught-green `#22C55E`** (reserved hit/caught semantic, already app-wide), on-color near-black text `#0C0C10`.

Accent reserved for: **Install / Refresh (SW update) prompts, the primary phase CTA (Share), the pre-opener "Search for the opener" prompt orb, focus rings, AND (new this phase) the bingo win badges — line / four-corners / X / blackout — ONLY.** Bingo win badges are the earned-payoff element of the replay surface and join the Share-CTA accent family; the accent stays a genuine <10% highlight. Never applied to all interactive elements.

**Phase-15 color contract (no new hex enters the codebase):**
- **Marked bingo square** → reuse the existing reserved **caught-green `#22C55E`** fill/border with near-black `#0C0C10` label text (the shipped on-color-orb contrast idiom, verified ≥ 4.5:1). A marked square is semantically "caught", so it inherits the caught-green already used for hits/dex.
- **Unmarked bingo square** → `#17171F` (elevated) with `#F5F5F7` label text and a `#2A2A34` hairline border.
- **Free center square** → marked treatment (caught-green) with a "Free" label; it is pre-marked (`FREE_SENTINEL`), never derived from the trail.
- **Win badge** → accent gold `#F2C14E` chip (small, restrained — the big celebration/supernova is Phase 16, NOT this read-only replay).
- **Rarity/tuning hues are untouched** — the separate `config.dex.tierColors` set (`#A1A1AA`/`#E4E4E7`/`#34D399`/`#60A5FA`/`#A855F7`/`#FB923C`) and the GizzVerse tuning-family palette are not introduced onto any bingo surface. Data semantics are never expressed in accent chrome.

All strings live in `config.copy.*` (new `config.copy.games.*` + additions to `config.copy.recap.*`), read by components — never hardcoded (the established convention). kglw-derived strings (square labels, song names, venue/city) render as **escaped React text only** (never `innerHTML` / `dangerouslySetInnerHTML`) — the T-06-21 discipline.

| Element | Copy |
|---------|------|
| Primary CTA (catch-up, BINGO-06) | **Catch me up** |
| Catch-up confirm-list submit | **Add {n}** (n = checked rows; `n === 1` → "Add 1 song") |
| Catch-up confirm-list heading | "Songs the tracker missed" |
| Catch-up confirm-list body | "Untick anything wrong, then add the rest to your setlist." |
| Catch-up manual affordance | "Search to add a song" (reuses the shipped fuse.js catalog search — D-04) |
| Catch-up nothing-to-add state | "You're all caught up — nothing to add." |
| Catch-up feed-unavailable (error) | "Couldn't reach the setlist feed. Search to add songs manually." (offline-safe fallback to the manual path — never a dead end) |
| GizzGames tab label | **GizzGames** (14px semibold, mirrors LiveGizz/GizzVerse/GizzDex) |
| GizzGames section heading | "GizzGames" |
| Deal teaser title | "Gizz Bingo" |
| Deal teaser body (D-02) | "Deal a bingo card before your next show." |
| Deal teaser affordance | disabled control labeled **"Coming soon"** (must read as intentional/forthcoming, never broken or errored) |
| Replay-list empty state heading | "No cards yet" |
| Replay-list empty state body | "Cards you play at a show will show up here to relive." |
| Replay section heading (in RecapView, D-05) | "Bingo" |
| Win badge labels (D-06) | "Line", "Four corners", "X", "Blackout" |
| Replay no-win state | "No lines this show" (board still renders full marks; section is still present — only absent when there is NO card) |
| Per-square "which song lit this" (D-06) | "Lit by {songName}" (14px muted caption on marked squares; free center shows "Free") |
| Reshuffle-lock explainer (guard lands this phase; greyed control is Phase 16) | "Locked at Start Show" |

**Destructive actions:** none introduced this phase. Catch-up **appends** to the trail (non-destructive); corrections use the already-shipped trail Undo/delete path (Phase 4 D-15) — no new confirmation dialog. Marking is derived; there is **no direct "tap a square to mark/unmark it"** (D-04), so no daub/undaub destructive surface exists.

**Copywriting rules for this phase:**
- The GizzGames empty state (empty replay list + "Coming soon" teaser) must feel **intentional, not broken** (D-02) — always render the teaser card + a friendly empty-list message, never a bare blank screen or an error.
- Catch-up is a **glance-and-correct** confirm-list, never a silent bulk-adopt — the pre-checked list plus "Untick anything wrong" copy keeps the human as the honest hit/miss denominator against a possibly mis-scraped feed (D-03).

## Interaction Contracts (phase-specific)

The load-bearing "interaction" half of the contract for the three new surfaces.

### D-01 — GizzGames: the 4th bottom tab
- **Contract:** add a 4th tab to `BottomTabBar.tsx` (`show` / `explore` / `dex` / **`games`**) and a `games` route to `useHashRoute.ts`. Tabs stay `flex-1` with `min-h-11 min-w-11`; active tab uses `text-accent`, inactive `text-text-muted` — reuse the exact shipped pattern. **Icon: `lucide-react` `Gamepad2`** (generic games hub — the tab is forward-compatible home for Gizzle / Guezz League per the deferred roadmap, so a bingo-specific icon like `Grid3x3`/`Dices` is deliberately avoided).
- **Acceptance:** four evenly-split tabs each ≥ 44px on the narrowest supported iPhone; the 3→4 tap-target tightening is accepted (D-01); the new tab routes to the GizzGames view.

### D-02 — GizzGames view: replay-only + honest teaser
- **Contract:** the view lists past shows that have a bingo card (each row → its RecapView) and renders a **"Gizz Bingo · Deal a card — Coming soon"** teaser card where the Phase-16 deal entry point will land. The teaser affordance is visibly **disabled**, not a live button.
- **Acceptance:** with zero cards the screen shows the teaser + "No cards yet" empty-list copy and reads as intentional/forthcoming — never blank, never an error, never a broken-looking control.

### BINGO-06 — Catch-up: grow the trail, never touch squares
- **Contract:** "Catch me up" presents a **pre-checked confirm-list** of `latest`-feed rows the tracker missed. On **Add {n}**, each checked row commits via the shipped `adoptSuggestion` path (`ShowView.handleAdopt`); manual additions reuse the fuse.js catalog search → `logSong` (miss). **No direct square tap** — every mark flows through the trail and `deriveMarks` re-lights squares as a pure consequence (D-03/D-04). Catch-up adds carry `shownFanSongIds: []` → classified as misses (honest denominator; consistent with `handleSearchSelect`).
- **Acceptance:** adding N songs via catch-up grows the trail by N and deterministically re-lights exactly the qualifying squares (live == replay == catch-up); the hit/miss tally stays honest (catch-up backfills count as misses, not phantom hits).

### BINGO-07 — Replay: read-only, pure re-derivation, full detail
- **Contract:** a past show's frozen card renders as a **"Bingo" section inside `RecapView`** — the final 4×4 board with marked squares, win badges (line / four-corners / X / blackout), and per-square "Lit by {song}" (D-06). Read-only. It is a **pure re-derivation** — `deriveMarks(card, trail, ctx, frozenCaughtSnapshot)` + `detectWins` over the persisted trail RecapView already reads — **marks are never stored** (D-23). The section is **absent** for shows with no card (D-05).
- **Acceptance:** the replayed board is byte-identical to what the user saw live over the same trail; a `neverCaught` square uses the FROZEN `caughtSnapshot` (a later show's catch never retroactively un-marks it); the opener square fires on the first real entry (0-based reindex adapter). Shows without a card show no Bingo section at all.

### Reshuffle lock (roadmap SC-1, D-10)
- **Contract:** reshuffle/draft-write is rejected (app-side write-helper invariant) when the target session is `finalized` or the card's `lockedAt != null`. The greyed reshuffle control + "Locked at Start Show" explainer is a **Phase-16** surface; Phase 15 lands the guard (the safety net) and the explainer copy.
- **Acceptance:** attempting a reshuffle on a locked/non-active session is hard-rejected; no bingo state leaks into `packages/core` (the guard is app-side; core stays DOM/DB-free).

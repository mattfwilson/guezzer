<objective>
Stand up the reactive session layer and the ShowView lifecycle so a user can Start Show (provisional attendance), and so the app write-throughs and restores an active session exactly. This plan proves the crash-proof half of the loop end-to-end (Start Show → write-through → force-quit → exact restore) and builds + integration-tests the tap-orb→log-hit→recenter wiring. The **first live opener seeding** (a real `currentSongId` via Search) and the on-screen first-recenter demonstration land in plan 04-05, because at Start Show there is no current song and `predict()` has nothing to rank.

Critical correctness rule (blocker fix): the frozen core type is `ShowContext.currentSongId: number` (non-nullable, packages/core/src/domain/types.ts:198). `null` must NEVER reach core. `currentSongId` is `number | null` in app state ONLY; the pre-opener state (`null`) renders the "Tap the opener" prompt and calls NO predict — no orbs render until a real song exists.

Purpose: Deliver SHOW-11 (write-through + exact restore) + DEX-01 (provisional attendance) as a genuinely reachable, tested slice, and wire SHOW-03 (tap→recenter) ready for its live demonstration once 04-05 seeds an opener.
Output: `useShowSession` (the app's first `useLiveQuery`), `ShowView` (with the null-safe pre-opener branch), `PreShowLauncher`, App/AppShell wiring, and the restore integration test.
</objective>

## Phase Goal (MVP user story)

**As a** KGLW showgoer, **I want to** start a show with one tap and trust the app to persist and restore my session exactly, **so that** a force-quit never loses my night's state or my dex credit — with the prediction loop becoming interactive the moment I log the opener (04-05).
